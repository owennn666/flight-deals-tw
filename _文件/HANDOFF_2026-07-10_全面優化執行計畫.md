### Phase 2 — 品質根治（含 schema 變更，動工前先向使用者求核准）

**P2-1 基準線分桶正確化**（後端，L；**P0 根治**，需 DDL 核准）
- 問題：baseline 把不同 gate（訂票網站）與不同出發期的價格混在同一池算分位，「比平常便宜」精度受汙染；且 prices 表根本沒存 gate（FarePrice.gate 入庫時被丟棄）。
- spec（已定案，照做）：
  1. DDL（先求核准）：`alter table public.prices add column if not exists gate text;` + `create index if not exists idx_prices_route_gate on public.prices (origin, destination, gate, observed_at);`
  2. `postgres_store.add_prices` INSERT 加 gate 欄（帶 `p.gate`）；memory/sqlite store 同步。
  3. `PriceStore.baseline` 簽名加 `gate: Optional[str] = None`：gate=None 維持現行為（全池，向下相容）；有值時只取該 gate 的觀測算分位，樣本門檻沿用 50+7 曆日（per 桶）。
  4. `pipeline.run_once`：對每航線先把 offers 依 gate 分組，逐 gate 取 `baseline(route, gate=g)` 判定該組 offers（gate 為 None 的 offer 用全池 baseline）。
  5. 過渡行為：舊 prices 列 gate=null 不入任何 gate 桶 → 各桶自 DDL 上線起重新累積約 7 天才可靠。**回報使用者時明講**：這會讓新好康再暖機一次。
  6. 「距出發天數」第二層分桶：**選配**，本輪不做（樣本密度不足），在程式註解留 TODO。
- 驗收：單元測試——同航線兩個 gate 各自的價分布不同，各自的 p10 判定互不汙染；上線後觸發 run 無錯、`select gate, count(*) from prices where observed_at > now() - interval '20 minutes' group by gate` 有分布。

**P2-2 降價舊卡下架 + 復現解噤聲**（後端+DDL 核准，M）
- 問題：同航線同日期出更低價時，舊的較貴 deal 還掛在列表；deals_seen 永久噤聲，復現的好康永遠不再觸發。
- spec（已定案）：
  1. DDL（先求核准）：`alter table public.deals add column if not exists superseded boolean not null default false;`
  2. `save_deal` 成功插入新列後，執行 `update deals set superseded = true where type=%s and origin=%s and destination=%s and depart_date is not distinct from %s and coalesce(gate,'') = coalesce(%s,'') and price > %s and id <> %s`。
  3. 前端 `api.deals()` query 加 `&superseded=is.false`。
  4. 解噤聲：`mark_deal` 改 `insert ... on conflict (key) do update set ts = now()`；`seen_deal` 改 `select 1 from deals_seen where key=%s and ts > now() - interval '45 days'`（45 天窗，復現可重推）。
  5. sqlite/memory store 同語意跟進（測試用）。
- 驗收：測試——插入同鍵更低價後舊列 superseded=true；seen_deal 對 46 天前的 key 回 False。上線後前端列表無「同航線同日期一貴一便宜並列」。

- spec：全 src 目前零筆 accessibility 屬性。至少補：訂票 CTA、票種/篩選 chips、清除鈕、篩選面板展開鈕、隱藏廉航 Switch 的 `accessibilityRole`/`accessibilityLabel`。
- 驗收：grep `accessibilityRole` 命中上述五類元件；tsc 零新增。

### Phase 4 — 安全根治（大工程，動工前先與使用者確認要不要現在做）

**P4-1 RLS ownership（Supabase Anonymous Auth）**（前端+DDL/policy 核准，L）
- 問題：subscriptions 匿名可讀全表（隱私）、devices 匿名可 update 任意列（防灌爆 CHECK 只擋大小不擋亂寫）。
- spec（方向定案，細節執行時再研究官方文件）：
  1. 前端加 `@supabase/supabase-js` 依賴，啟用 Anonymous sign-in（Supabase 專案設定要開 anonymous provider——使用者或 Management API 開）；client.ts 改帶 session JWT 打 PostgREST；device 識別改用 `auth.uid()`。
  2. RLS 重寫（先列 SQL 求核准）：devices/subscriptions 全部 policy 改 `device = auth.uid()::text`（或加 owner 欄）只准操作自己那列；移除 anon 寬鬆 policy；deals 匿名讀不變。
  3. 遷移：既有匿名 UUID 訂閱對不上新 auth uid，量極少，接受使用者重存一次；舊列由使用者 SQL Editor 清（不代刪）。
- 驗收：兩個瀏覽器 profile 各存一筆訂閱後，curl 各自 JWT 只讀得到自己那筆；匿名 session 重新整理後仍持久。
