# 便宜機票偵測平台 — 產品與技術架構文件

> 定位：台灣出發、整合「四腳票 / bug 票 / 異常便宜票」三種票源的自動化偵測與推播平台
> 開發方式：AI 輔助開發（Claude Code / Cursor 等）
> 商業策略：先衝流量，後變現（聯盟行銷起步）
> 前端：React Native (Expo) App（iOS/Android）；後端：Python FastAPI（保留偵測引擎當 API）
> 版本：v2（RN 版）　日期：2026-07-07

---

## 0. TL;DR（一頁看懂）

- **做什麼**：一個「幫台灣人自動抓便宜機票」的平台。核心不是賣票，而是**偵測 + 推播**，把訂票導流到 OTA / 航司官網賺聯盟佣金。
- **三種票的本質不同，技術上要分開處理**：
  - **異常便宜票** = 統計上偏低但合法的票 → 靠「價格基準線 + 異常偵測」自動抓。
  - **bug 票 (error fare)** = 航司標錯價 → 靠「極端異常 + 人工複核」，拚速度（可能只存活幾分鐘）。
  - **四腳票（外站四段票）/ 隱藏城市票** = 靠「構票演算法」組合多段航程壓低票價 → 需要 multi-city 搜尋能力。
- **MVP 最短路徑**：先用 Travelpayouts（免費 + 內建聯盟變現）+ Amadeus 免費額度，鎖定 10–20 條台灣熱門航線，做「異常便宜票」自動偵測 → LINE 推播 → 導聯盟連結。**先驗證需求，再往 bug 票、四腳票擴充。**
- **法律關鍵**：把自己定位成**「資訊分享 / 比價」而非「開票代理」**，訂票一律導外部，可大幅降低責任。隱藏城市/棄程「違約但不違法」，一定要放免責聲明。
- **前端架構（本版重點）**：客戶端用 **React Native (Expo) 跨平台 App**（一套碼上 iOS/Android），後端保留 **Python FastAPI** 偵測引擎當 API。App 是「薄客戶端」：所有偵測留在伺服器端跑，App 只負責看好康、管訂閱、收推播。
- **台灣特色**：**推播走 App 原生 push（Expo Notifications）＋ LINE 雙管道**，不是 email/WhatsApp。競品「布萊N」證明台灣人願意付 NT$168/月訂便宜機票情報。

## 1. 三種票種：定義與技術含義

搞清楚三種票的**本質差異**，因為它們的偵測邏輯完全不同，這是整個系統設計的地基。

### 1.1 異常便宜票（Abnormally Cheap Fare）
- **定義**：票價「合法、航司故意促銷或淡季甩賣」，但明顯低於該航線的歷史／季節均價。例如平常 TPE→東京來回 NT$12,000，突然出現 NT$6,500。
- **技術含義**：需要「**每條航線的價格基準線**」，用統計方法判斷「便宜多少算異常」。這是最容易自動化、誤報成本最低的一種。
- **存活時間**：數小時到數天，壓力不大。

### 1.2 bug 票 / error fare（標錯價票）
- **定義**：航司或系統**出錯**導致的離譜低價。常見成因：人為打錯數字、幣別換算 bug、fare class 載入錯誤、燃油附加費漏算。例：頭等艙賣經濟艙價、跨洲來回 NT$5,000。
- **技術含義**：屬於「**極端異常**」，門檻要拉很高（例如低於基準線 80%+，或艙等/票價比明顯錯亂），且**必須交叉比對多來源 + 人工快速複核**再推播，否則誤報會嚴重傷害信任。
- **存活時間**：**幾分鐘到幾小時**，速度是生命線。這是最難、但也最有話題性、最能衝流量的一種。

### 1.3 四腳票 / 外站四段票（Nested / Outer-Station Ticket）＋ 隱藏城市票
- **四段外站票定義**：以**非台灣（外站）為起點**、台灣為中停點的多段票。典型結構是「外站 → 台北 → 外站」共四個航段，可分兩趟旅程用，長榮 / 華航 / 星宇這類有第五航權或外站定價優勢的航司常見。實務關鍵：票有**順序性**，第一段 No-show 會導致後續三段全被系統取消；棄掉「最後一段」通常較安全。
- **隱藏城市票（Hidden City）定義**：買 A→C（中停 B）的票，實際只飛到 B 就下機，因為 fare(A→C) < fare(A→B)。
- **技術含義**：這兩種都不是「等便宜出現」，而是「**主動構票**」——用 multi-city 搜尋去組合出比直購更便宜的航段組合，需要能做多城市 / 開口程搜尋的資料源（Kiwi Tequila、Amadeus multi-city）。

> **設計原則**：三種票共用同一套「資料擷取 + 儲存 + 通知」基礎設施，但**偵測引擎要拆成三個獨立模組**，各自有不同門檻與驗證流程。

1. **台灣出發航線最佳化**（TPE/RMQ/KHH），航線、幣別、簽證脈絡都在地化。
2. **三種票整合在一個平台**（多數在地玩家只做人工分享單一類型）。
3. **自動化偵測 + 即時性**（機器掃描 vs 人工發現）。
4. **LINE 為核心推播管道**（台灣滲透率最高，比 email/WhatsApp 更貼近本地）。

> **關鍵界線**：輪詢與偵測是「常駐後端」的工作，RN App 不跑這些；App 只透過 FastAPI 讀好康、管訂閱、收 push。這就是為什麼保留 Python 後端。

## 4. 資料來源策略（最關鍵的技術決策）

> **核心觀念**：沒有任何單一 API 能同時給你「即時全網最低價 + 便宜的價格 + 構票能力」。務實做法是**分層混用**，並優先用「有聯盟授權的官方 API」以同時解決**資料 + 變現 + 法律**三件事。

### 4.1 主要 API 比較

| API | 免費額度 | 計費 | 強項 | 弱項 | 建議角色 |
|---|---|---|---|---|---|
| **Travelpayouts / Aviasales（v3）** | **免費**（免費註冊拿 token）| 免費，靠聯盟 marker 抽佣 | **免費 + 內建變現**、近 48h 最低價、涵蓋台灣航線 | 為搜尋快取（非即時、存 2–7 天）| **免費首選 · 本版已實作** |
| **SerpApi — Google Flights** | 250 次/月 | 超額付費（每次查詢計費）| 直接拿 Google Flights 結果 + price insights | 免費額度很小、成本高 | 需要時做 bug 票交叉驗證 |
| **Duffel** | Starter 免費（每月 50 筆訂單）| 每張確認訂單 $3 | 可真的線上開票、現代 API | 以「賣票」為核心，非比價 | 之後若要自營訂票再考慮 |
| **Amadeus Self-Service** | —（新註冊已關）| — | 正規 GDS、multi-city | ⚠️ **2026/7/17 停用，勿用於新專案** | 已淘汰 |
| **Kiwi.com Tequila** | 需申請 | 聯盟 / 賣票 | 虛擬聯程、構票能力最強 | ⚠️ **需 5 萬 MAU 才給存取** | 新專案拿不到 |

> **2026 免費 API 現況（重要）**：能「真正免費 + 有價格 + 涵蓋台灣」的，實務上就 **Travelpayouts** 一個（本版已實作 v3 `prices_for_dates`）。Amadeus 2026/7/17 停用、Kiwi 需 5 萬 MAU、Google Flights 無官方免費 API。沒有「免金鑰」的優質價格 API，但 Travelpayouts 只需免費註冊拿 token，且 token 同時是聯盟變現的 marker 來源。

### 4.2 建議的資料源組合（依階段）
- **MVP（Phase 1）**：**Travelpayouts** 掃全航線抓異常便宜票 + 聯盟 marker 變現（單一免費來源就能起步，本版已可接）。
- **加 bug 票（Phase 2）**：命中極端低價時，用 **SerpApi**（小額付費）或第二來源做交叉驗證，確認不是快取雜訊再推播。
- **加四腳票（Phase 3）**：構票需 multi-city 資料——Amadeus 已停用、Kiwi 門檻高，屆時可能得評估付費 API 或自建查詢，此為**已知待解問題**。

### 4.3 關於「爬蟲」的務實建議
自己爬 Google Flights / Skyscanner / 航司官網技術上可行，但：**（1）** 對方有反爬蟲（需住宅代理、session 一致性，維運成本高）；**（2）** 可能違反對方 ToS，有法律風險。
**建議**：MVP 階段**不要自建爬蟲**，優先用官方 API + 聯盟資料源。等規模與收入起來、確認某條資料缺口非爬不可時，再局部評估。

## 5. 三種票的偵測演算法（系統的核心 IP）

### 5.1 異常便宜票：價格基準線 + 異常偵測

**目標**：對每條航線（origin–destination–月份）維護一條「正常價格基準線」，判斷當前價是否顯著偏低。

**步驟**：
1. **建基準線**：對每條航線持續蒐集歷史價格，計算「trailing 30/90 天的中位數與分佈」，並做**季節性調整**（過年、暑假、連假本來就貴，不能一視同仁）。
2. **異常分數**：用穩健統計，例如
   - 百分比法：`當前價 < 基準中位數 × (1 − θ)`，θ 先設 25–35%。
   - 穩健 z-score：`(median − price) / MAD`（用中位數絕對差，抗離群值）。
3. **過濾誤報**：需連續 N 次抓到同一低價、或多來源一致，才算數（避免單次快取雜訊）。
4. **分級**：便宜 25–40% = 一般好康；40–60% = 強力推薦。

**儲存**：價格時序放 **PostgreSQL + TimescaleDB**（或 ClickHouse）。MVP 用 Timescale 就夠，AI 也最熟。

### 5.2 bug 票 / error fare：極端異常 + 多源交叉 + 人工複核

**目標**：抓出「離譜到幾乎確定是標錯」的價，但把誤報壓到最低。

**步驟**：
1. **極端門檻**：低於基準線 **70–85%+**，或觸發結構性異常規則：
   - 艙等/票價錯亂（商務/頭等艙賣經濟艙價）。
   - 票價/距離比異常（長程洲際票出現短程票價）。
   - 含稅總價 < 單純稅金與燃油附加費的合理下限。
2. **多源交叉驗證**：命中後**立刻**用第二、第三來源（SerpApi / Amadeus）重查，三方都便宜才可信 → 大幅降低「假 bug」。
3. **人工複核關卡（human-in-the-loop）**：進「待審佇列」，你或審核者在後台一鍵「確認/駁回」。error fare 誤報一次，用戶信任崩很快。
4. **搶速度**：從偵測到推播要壓在**秒~分鐘級**。用訊息佇列 + 非同步推播（見第 6 章）。
5. **推播內容務必含免責**：「疑似標錯價、航司不保證出票、訂票風險自負、建議先別訂旅館」。

### 5.3 四腳票 / 隱藏城市：構票演算法

**四段外站票（構票）**：
- 對目標旅程（例如「想飛歐洲」），用 multi-city 搜尋去組合「外站 → TPE → 目的地 → 外站」的四段組合，比較「構出來的四段票總價」vs「TPE 直購來回」。
- 命中條件：`構票總價 < 直購價 × (1 − θ)`，且航段時間可銜接、同一航司/聯盟可開同一票。
- **一定要在結果標註風險**：航段順序性（第一段 No-show → 後段全取消）、行李政策、改簽限制。

**隱藏城市（Hidden City）**：
- 對 A 出發、想到 B：搜尋所有「A → B → C（C 為更遠的第三地）」且 `fare(A→C 中停B) < fare(A→B)` 的組合。
- **風險標註（必須）**：只能買單程、不能託運行李到最終目的、航司可能取消回程或封鎖帳號、常客里程可能被追討。

> **這一塊的差異化最高、但法律/ToS 風險也最高**，建議放在 Phase 3，且以「教育 + 資訊呈現」方式提供，不代訂。

## 6. 技術選型（為 AI 輔助開發最佳化）

> 選型原則：**選主流、社群大、訓練資料多的技術**，這樣 Claude Code / Cursor 生成與除錯的品質最高、你踩坑最少。

| 層 | 建議 | 理由 |
|---|---|---|
| **後端 / 偵測** | **Python 3.12 + FastAPI** | 保留偵測引擎；FastAPI 同時當 App 的 REST API，AI 最熟 |
| **App（客戶端）** | **React Native + Expo (TypeScript)** | 一套碼跨 iOS/Android；Expo 內建 push/OTA 更新，AI 生成品質高 |
| **落地頁（選配）** | **Next.js** | App 之外補一頁 SEO 落地頁導自然流量，非必要 |
| **資料庫** | **PostgreSQL + TimescaleDB** | 一個 DB 兼顧關聯資料 + 價格時序；免多養一套 |
| **快取 / 佇列** | **Redis** | 即時去重、rate limit、輕量任務佇列 |
| **排程** | Cron / **APScheduler** / Cloud Scheduler | 定時 polling |
| **非同步任務** | **Celery**（或 RQ）| bug 票搶速度、通知扇出 |
| **推播** | **Expo Notifications（App 原生 push）+ LINE Messaging API** | App 用戶走原生 push；LINE 觸及更廣社群 |
| **部署** | App 用 **Expo EAS**（送 App Store / Play）；後端 **Railway / Render / Fly.io**；選配落地頁 Vercel | 起步便宜、部署簡單 |
| **監控** | Sentry（錯誤）+ 簡單健康檢查 | 早期夠用 |

**倉庫結構建議（monorepo）**：
```
/apps
  /mobile        # React Native (Expo) App — 客戶端（本版新增）
  /web           # Next.js SEO 落地頁（選配）
/backend         # Python：偵測引擎 + FastAPI
  /flightdeals
    /ingest      # 各資料源 API 客戶端 + 正規化
    /detectors   # 三個偵測模組（cheap / errorfare / nested）
    /storage     # 時序 + 基準線 + deals 持久化
    /notify      # Expo push / LINE / console
    /api         # FastAPI：/deals /routes /subscriptions /devices
/infra           # 部署設定、排程
```
> 模組邊界明確，很適合把每個資料夾當成一個「給 AI 的獨立任務」逐步開發。
> 本版已附可執行骨架：`backend`（Python 偵測引擎 + FastAPI）與 `apps/mobile`（Expo App）。

## 7. MVP 範圍與開發路線圖（先衝流量版）

### Phase 0 — 半自動驗證需求（1–2 週，先別寫一堆程式）
- 手動 + 簡單腳本，盯 **10–20 條台灣熱門航線**（東京、大阪、首爾、曼谷、峴港、歐洲主要城市）的異常便宜票。
- 建一個 **LINE 官方帳號 + FB 社團**，發現好康就發，訂票連結先手動掛聯盟。
- **目的**：驗證「有沒有人要看、哪種票最有感」，同時開始養第一批粉絲。這一步就是複製「布萊N」早期模式，但你之後會自動化。

### Phase 1 — 自動化「異常便宜票」（MVP 核心，約 3–6 週）
- Ingestion：Travelpayouts 掃航線；Amadeus 免費額度複查。
- Detection：異常便宜票基準線 + z-score（第 5.1 節）。
- 儲存：Postgres + TimescaleDB。
- 通知：Expo 原生 push + LINE（可依航線/預算訂閱）。
- App：React Native (Expo) 好康列表 + 訂閱設定 + 原生 push；選配 Next.js 落地頁做 SEO。
- 變現：所有訂票按鈕掛 **Travelpayouts / Trip.com / Skyscanner 聯盟連結**。

### Phase 2 — 加「bug 票 / error fare」（約 4–6 週）
- 極端異常門檻 + 多源交叉驗證（SerpApi/Amadeus）。
- 後台「人工複核佇列」+ 一鍵發布。
- 搶速度：Celery 非同步 + LINE 即時推播。
- 這是**話題性最高、最能爆流量**的功能。

### Phase 3 — 加「四腳票 / 隱藏城市」構票（約 6–8 週）
- 導入 Kiwi Tequila / Amadeus multi-city。
- 構票演算法 + 完整風險標註 + 教學內容。
- 定位為「進階省錢工具 + 教育」，不代訂。

### 之後的變現升級（流量夠了再做）
- 訂閱制（早鳥推播、進階篩選）——布萊N 的 NT$168/月已驗證付費意願。
- 聯盟佣金優化、與 OTA 談客製合作。

## 8. 法律與 ToS 風險（務必先讀）

> 我不是律師，以下為根據公開資訊的整理，正式上線前請諮詢台灣執業律師。

### 8.1 平台定位＝最重要的風險控制
**把自己定位成「資訊分享 / 比價媒體」，不自營開票、訂票一律導向 OTA 或航司官網（聯盟連結）。** 這樣你賣的是「資訊與流量」，不介入運送契約，法律曝險最低。這也剛好符合你「先衝流量、聯盟變現」的策略。

### 8.2 隱藏城市 / 棄程 / 四腳票
- 這類操作通常**「違反航司運送契約，但不違法」**。風險落在**旅客**身上（航司可取消後續航段、追討里程、封鎖帳號）。
- 判例參考：Skiplagged 被 American Airlines 告，2024/10 判賠 **940 萬美元**——但關鍵是**主因為著作權/商標，非「隱藏城市」本身被判違法**；AA 原求償 9,440 萬，且商標主張未獲賠。更早 United（2014）、Southwest（2021）對類似業者的訴訟多被駁回。
- **對你的意涵**：提供資訊/教學風險相對可控，但**務必顯著揭露風險與免責**，且不要「代客操作訂票」。

### 8.3 bug 票 / error fare
- 台灣消保觀點：**買賣契約一旦成立，航空公司原則上應履行**；但**個案認定**，且「透過社群瘋搶、明顯投機」可能影響受保護程度。
- **境外航司**在台無據點時，爭議協調困難，往往只能自行對外聯繫。歷史上有航司自行認賠案例（如華航 2011、國泰 2018）。
- **對你的意涵**：推播 bug 票時**明確標註「疑似標錯價、不保證出票、風險自負、建議先別訂不可退的旅館/行程」**，把期待管理好，保護用戶也保護自己。

### 8.4 資料抓取 / API ToS
- 優先用**官方 API + 聯盟授權**，避免違反 Google Flights / 航司官網的 ToS。
- 遵守各 API 的 rate limit 與快取/顯示條款（例如 Amadeus、Travelpayouts 對資料使用的規範）。

### 8.5 上線前檢查清單
- [ ] 網站放「服務條款 + 免責聲明 + 隱私政策」。
- [ ] bug 票 / 隱藏城市頁面有顯著風險揭露。
- [ ] 個資（LINE ID、email、追蹤偏好）符合台灣《個資法》。
- [ ] 聯盟揭露（affiliate disclosure）。
- [ ] 諮詢律師確認平台定位與免責措辭。

## 10. 關鍵風險與注意事項（Reality Check）

1. **資料即時性 vs 成本的拉扯**：免費資料源多為快取（非即時），但即時 API 貴。bug 票要搶快，這個矛盾是產品成敗核心——先用「便宜資料掃全網、命中才用貴資料複查」的兩段式設計省錢。
2. **誤報是信任殺手**：尤其 bug 票，寧可慢、不可錯。人工複核關卡別省。
3. **航線基準線需要時間累積**：剛上線沒有歷史資料，前 1–2 個月偵測會不準，可先用第三方價格趨勢資料 bootstrap。
4. **推播疲勞**：太多通知會被封鎖/退訂。讓用戶精準訂閱（航線/預算/艙等）。
5. **法律灰色地帶**：三種票中，隱藏城市/四腳票爭議最大，建議晚點做、以教育定位、務必免責。
6. **反爬蟲與 API 政策變動**：資料源政策可能說變就變，架構上讓「資料源」可插拔（第 6 章的 `/ingest` 模組化）以降低單一來源風險。

## 11. 前端與 API 架構（React Native + Python API）

### 11.1 為什麼是「RN App + Python API」而不是把後端也改 JS
- **RN 是客戶端，不是後端**：輪詢多個航班 API、維護基準線、跑偵測、扇出推播，這些是**常駐伺服器**的事，手機 App 不適合也不該做。
- **保留 Python 後端**＝重用已寫好、已測過的偵測引擎，且資料/統計/未來 ML 異常偵測的生態是 Python 最強。
- **RN (Expo) 的好處**：一套 TypeScript 碼同時上 iOS/Android **與 Web（react-native-web）**；Expo 內建 push、OTA 熱更新、EAS 打包上架；社群大、AI 生成品質高。
- **同一套碼三平台**：`npx expo start` 開 App、`npx expo start --web` 直接跑成網頁——所以你不用另外寫一份 web 前端；等要做 SEO 落地頁再選配 Next.js。

### 11.2 分工界線（很重要）
| 放在後端（Python/FastAPI） | 放在 App（React Native/Expo） |
|---|---|
| 資料擷取、基準線、三種票偵測、去重、人工複核 | 看好康列表/詳情、篩選、下拉刷新 |
| 存 deals、管理訂閱與 push token | 設定訂閱條件（航線/預算/艙等）|
| 決定「誰該收到哪筆好康」並送出推播 | 註冊 Expo push token、收原生通知、點擊深連結去 OTA |

### 11.3 API 合約（FastAPI，本版已附骨架）
| 方法 | 路徑 | 用途 |
|---|---|---|
| `GET` | `/api/health` | 健康檢查 |
| `GET` | `/api/routes` | 可追蹤的航線清單 |
| `GET` | `/api/deals?limit=&type=` | 近期好康（App 首頁列表；可依票種篩選）|
| `GET` | `/api/deals/{id}` | 單筆好康詳情（含訂票聯盟連結）|
| `POST` | `/api/devices` | 註冊/更新 Expo push token（裝置）|
| `GET` / `POST` | `/api/subscriptions` | 查詢/設定用戶訂閱條件（航線、預算、艙等）|

回應一律 JSON，App 端有對應的 TypeScript 型別（`apps/mobile/src/api/types.ts` 與後端 `schemas.py` 對齊）。

### 11.4 推播流程（Expo push + LINE）
1. App 啟動向使用者要通知權限 → 拿到 **Expo push token** → `POST /api/devices` 存到後端。
2. 偵測引擎命中好康 → 依「訂閱條件」找出該收到的裝置 → 打 **Expo Push API** 送原生通知；同時（選配）LINE broadcast 給更廣社群。
3. 用戶點通知 → 開 App 的好康詳情 → 點「訂票」開外部瀏覽器（**聯盟連結**，平台不代訂）。

> **本版已實作此閉環**：`flightdeals/subscribers.py`（裝置/訂閱持久化 + 好康比對）＋ `flightdeals/notify/expo_push.py`（打 Expo Push API），以 `expo_push` 註冊進 registry，設定檔啟用即可。已有測試驗證「只推給訂閱條件命中的裝置」。

### 11.5 App 畫面（骨架已含）
- **好康列表 (Deals)**：即時抓 `/api/deals`，卡片顯示票種徽章、省幾成、價格。
- **好康詳情 (DealDetail)**：完整資訊 + bug 票風險免責 + 訂票（聯盟）按鈕。
- **訂閱設定 (Subscriptions)**：選航線 / 預算 / 艙等，寫回 `/api/subscriptions`。
- **設定 (Settings)**：推播開關、免責與隱私說明。

### 11.6 一套碼跑三平台（iOS / Android / Web）
Expo 透過 **react-native-web** 讓同一份畫面碼直接編譯成網頁：
- `npx expo start` → 手機 App（Expo Go / 模擬器）
- `npx expo start --web` → 瀏覽器網頁（同樣的 `App.tsx` 與畫面）

所以「App」與「基本 web 前端」是同一套程式，不用維護兩份。等到需要**行銷/SEO 落地頁**（衝自然流量）時，再選配獨立的 Next.js 頁面即可；產品主體維持 RN 一套碼。
